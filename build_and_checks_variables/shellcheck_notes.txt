A listing of the current suppressed warnings/errors from shellcheck,
and also some of the warnings that were suppressed in some previous
state of the git repository (in that case, we use X suffix, see SC2162X):
- SC1090 : Can't follow non-constant source.
           Use a directive to specify location
           # shellcheck source=bla/bli.sh
           Only remaining cases are in common_build_and_checks
           when sourcing some Python venv
           source venv/bin/active
- SC1091 : Not following (file not found most of the time).
           Only remaining cases are in common_build_and_checks
           when sourcing some Python venv
           source venv/bin/active
- SC1111 : Use of slanted double quotes (valid inside single quotes)
           False positive, but may catch some problem.
- SC2001 : Replace some sed with bash builtin ${a//b/c}
           False positive when c contains \n.
- SC2016 : Expressions don't expand in single quotes,
           use double quotes for that.
- SC2034 : Unused variable
           False positive for globals that are declared in functions
           that are not currently called.
           Somehow it can be used to detect dead code.
- SC2059 : Don't use variables in the printf format string.
           Use printf '..%s..' "$foo".
           False positive when using a variable to build the format string.
- SC2086 : Double quote to prevent globbing and word splitting.
           False positive when returning integer variable
           via an indirection like return ${!foo}.
           See SC2248 below and the corresponding issue on GitHub.
- SC2119 : Use foo "$@" if function's $1 should mean script's $1.
           False positive when passing arguments would yield
           a wrong result.
- SC2120 : foo references arguments, but none are ever passed.
           False positives when some arguments are meant for debug in CLI.
- SC2125 : Brace expansions and globs are literal.
           False positive when assigning a variable declared with -i.
           https://github.com/koalaman/shellcheck/issues/914
- SC2145 : Argument mixes string and array.
           False positive when using "$@" line return and pipes.
           https://github.com/koalaman/shellcheck/issues/3192
           Example:
             # shellcheck disable=SC2145
             all_self_code_lines "$@"\
               | empty_lines_after_file_name
- SC2154 : var is referenced but not assigned.
           Very useful but false positive when I play with
           variable variable names.
- SC2162X: read without -r will mangle backslashes.
           False positive when read is used without variable:
           https://github.com/koalaman/shellcheck/issues/623
- SC2248 : Prefer double quoting
           False positive on returning integer values.
           https://github.com/koalaman/shellcheck/issues/3405
- SC2309 : -eq treats this as a variable.
             Use = to compare as string (or expand explicitly with $var).
           Very useful but false positive when I play with
           variable variable names.
- SC2312 : Some command is not invoked separately and thus its return
           value is masked. Frequent with pipes.
- SCNNNNX: Use printf instead of echo.
           https://github.com/koalaman/shellcheck/issues/3419

The command to check if there is anything else:
grep -r "shellcheck disable" . | grep -v -e '.git' -e "=SC1090$"\
  -e "=SC1091$" -e "=SC1090,SC1091$" -e "=SC1111$" -e "=SC2001$"\
  -e "=SC2016$" -e "=SC2034$" -e "=SC2059$" -e "=SC2086$" -e "=SC2119$"\
  -e "=SC2120$" -e "=SC2125$" -e "=SC2145$" -e "=SC2154$" -e "=SC2248$"\
  -e "=SC2309$" -e "=SC2312$" -e "=SC2154,SC2309$"
